# Use Case 1: Fetch existing DTSK work_notes

## Purpose

Fetch the existing `work_notes` for each DTSK before attempting any update.

This is required to avoid duplicate work_notes when the workflow is rerun.

## ServiceNow source

Table:

```text
sys_journal_field
```

Filter logic:

```text
element_id = <DTSK sys_id>
element    = work_notes
```

Where:

```text
<DTSK sys_id> = sys_id from x_alfi_decom_decom_task
```

Example query pattern:

```text
/sys_journal_field?sysparm_query=element_id=<DTSK_SYS_ID>^element=work_notes
```

## Expected result

The workflow should collect existing journal entries for the DTSK and search for a workflow marker.

Suggested marker:

```text
[COHESITY_BACKUP_VALIDATION_WORKFLOW]
```

## Duplicate prevention rule

If an existing work_note already contains the workflow marker for the DTSK and CI, the workflow must not write the same note again.

# Use Case 2: Update DTSK work_notes

## Purpose

Write a validation note back to the DTSK after backup validation.

## ServiceNow target

Table:

```text
x_alfi_decom_decom_task
```

Target record:

```text
sys_id = <DTSK sys_id>
```

Field to update:

```text
work_notes
```

## Writeback rule

Do not update `sys_journal_field` directly for normal work_notes writeback.

Use the DTSK table record and update the `work_notes` field on the DTSK record.

## Suggested work_notes format

```text
[COHESITY_BACKUP_VALIDATION_WORKFLOW]
Backup validation completed for CI: <CI_NAME>
Validation result: <RESULT>
Server-level backup found: <YES/NO>
Database backup found: <YES/NO>
Backup types found: <FS/VM/HyperV/Nutanix/SQL/Oracle>
Report rows: <ROW_COUNT>
Generated by Dynatrace workflow.
```

## Idempotency rule

Before writing work_notes:

```text
1. Fetch sys_journal_field where element_id = DTSK sys_id and element = work_notes
2. Search for [COHESITY_BACKUP_VALIDATION_WORKFLOW]
3. If marker exists for same DTSK/CI, skip work_notes update
4. If marker does not exist, update DTSK work_notes
```

# Use Case 4: Change DTSK state

## Purpose

Move the DTSK through the correct lifecycle after backup validation.

## Target table

```text
x_alfi_decom_decom_task
```

Target record:

```text
sys_id = <DTSK sys_id>
```

Field:

```text
state
```

## Known target values

```text
2 = Work in Progress
3 = Closed Complete
```

## Proposed logic

### When backup is found

If backup validation confirms an in-scope backup exists, set state to:

```text
2 = Work in Progress
```

Then, when all required update fields and work_notes are complete, set state to:

```text
3 = Closed Complete
```

## Important condition

Only close complete when backup is found and the update rules are satisfied.

Do not close complete DTSKs where:

```text
NoObject
Validation Error
Missing required ServiceNow fields
Assignment failed
work_notes update failed
```

# Use Case 6: Assign DTSK to a shift owner

## Purpose

Assign the DTSK to a person in the backup shift instead of leaving it unassigned.

## Required ServiceNow objects

Potential tables:

```text
sys_user
sys_user_grmember
sys_user_group
```

## Required input placeholders

```text
<BACKUP_GROUP_NAME>
<BACKUP_GROUP_SYS_ID>
<SHIFT_USER_FILTER>
```

## User lookup option 1: userLIKE search

Search users using a user/display-name filter.

Example pattern:

```text
/sys_user?sysparm_query=nameLIKE<USER_SEARCH_TEXT>^active=true
```

Use this only if a specific user search string is provided.

## User lookup option 2: group membership search

Preferred approach for round-robin assignment.

Step 1: Find group sys_id.

```text
/sys_user_group?sysparm_query=name=<BACKUP_GROUP_NAME>
```

Step 2: Get active users in group.

```text
/sys_user_grmember?sysparm_query=group=<BACKUP_GROUP_SYS_ID>
```

Step 3: Resolve user sys_id/display name from the member records.

Step 4: Assign DTSKs in round-robin order.

## Round-robin rule

Example:

```text
DTSK 1 → User 1
DTSK 2 → User 2
DTSK 3 → User 3
DTSK 4 → User 1
```

## Assignment target field

```text
assigned_to = <USER_SYS_ID>
```

## Fallback rule

If no eligible user is found:

```text
Do not update assigned_to
Add warning to workflow output
Do not close complete automatically
```

# Use Case 7: Update only DTSKs where backup is found

## Purpose

Avoid changing DTSKs where the workflow cannot confirm backup protection.

## Backup found definition

Backup is found when at least one in-scope backup row exists with one of these backup types:

```text
FS
VM
HyperV
Nutanix
SQL
Oracle
```

## No update condition

Do not perform writeback when the result contains only:

```text
NoObject
Unknown
Validation Error
```

## DB-only condition

If SQL or Oracle is found but no server-level backup is found, the workflow should not automatically close complete unless the process owner approves this behavior.

Current marker:

```text
NoFSBackupFound
```

Meaning:

```text
Database backup found, but no FS/VM/HyperV/Nutanix server-level backup found.
```

Recommended action:

```text
Update work_notes only, do not close complete automatically until policy is confirmed.
```

# Proposed writeback decision matrix

| Validation result | Work notes | Assign to user | CR Required = No | State = 2 WIP | State = 3 Closed Complete |
|---|---:|---:|---:|---:|---:|
| FS/VM/HyperV/Nutanix found | Yes | Yes | Yes | Yes | Yes |
| SQL/Oracle found with server-level backup | Yes | Yes | Yes | Yes | Yes |
| SQL/Oracle found only, NoFSBackupFound | Yes | Optional | No/Confirm | Yes | No |
| NoObject | Yes | No | No | No | No |
| Validation Error | Yes | No | No | No | No |

---

# Proposed writeback sequence

```text
1. Read aggregate validation result
2. For each DTSK result, decide writeback eligibility
3. Fetch existing work_notes from sys_journal_field
4. Check marker to avoid duplicate update
5. If eligible, identify assignment user
6. Update DTSK work_notes
7. Update assigned_to if eligible user exists
8. Update backup status field when field name is confirmed
9. Set CR Required = No when field name is confirmed and result is eligible
10. Set state = 2 WIP
11. Set state = 3 Closed Complete only when all required writebacks succeed
```

# Implementation guardrails

## Do not mix validation and writeback

Cohesity validation should remain separate from ServiceNow writeback.

Recommended separation:

```text
dtsk_validate_one_ci       = GET-only validation
dtsk_aggregate_report      = report building
dtsk_send_email_report     = notification
dtsk_update_servicenow_status = future writeback
```

## Idempotency is mandatory

The writeback task must be safe to rerun.

Minimum duplicate prevention marker:

```text
[COHESITY_BACKUP_VALIDATION_WORKFLOW]
```

## Do not close uncertain records

The workflow must not close complete any DTSK unless backup validation is confirmed and required writeback fields are successfully updated.

## Keep placeholders until confirmed

Leave these values configurable until ServiceNow fields are confirmed:

```text
<BACKUP_GROUP_NAME>
<BACKUP_GROUP_SYS_ID>
<SHIFT_USER_FILTER>
<BACKUP_STATUS_FIELD>
<CR_REQUIRED_FIELD>
```
