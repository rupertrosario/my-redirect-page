# DTSK Send Email Report Task

## Purpose

This task sends the output from `dtsk_aggregate_report` by email.

It does not call ServiceNow or Cohesity directly. It only uses the Markdown report generated by the previous aggregate task.

## Workflow position

```text
dtsk_snow_search
→ dtsk_prepare_work_items
→ dtsk_get_cluster_map
→ dtsk_validate_one_ci      [LOOP]
→ dtsk_aggregate_report     [NO LOOP]
→ dtsk_send_email_report    [EMAIL TASK]
```

## Task name

```text
dtsk_send_email_report
```

## Task type

Use the Dynatrace email/send notification action.

Do not use a loop for this task.

## Subject

Use dot notation because this Dynatrace email action does not accept bracket notation.

```jinja
Cohesity Backup Validation - Decommission DTSKs - {{ result("dtsk_aggregate_report").summary.totalDtsks }} DTSKs / {{ result("dtsk_aggregate_report").summary.totalRows }} Rows
```

Example:

```text
Cohesity Backup Validation - Decommission DTSKs - 15 DTSKs / 28 Rows
```

## Body

```jinja
{{ result("dtsk_aggregate_report").markdown }}

---

Note:
This report is generated using GET-only validation from ServiceNow and Cohesity Helios. No ServiceNow DTSK updates, state changes, assignment changes, CR Required changes, or work_notes updates are performed by this workflow.

Backup validation includes FS, VM, Hyper-V, Nutanix/AHV, SQL, and Oracle objects. NAS backups are intentionally excluded from this report.

NoFSBackupFound means a SQL or Oracle database backup was found, but no server-level backup was found for the same CI. Server-level backup means FS, VM, Hyper-V, or Nutanix/AHV.

NoObject means no in-scope Cohesity backup object was found for the CI.

NoBackupTime means a backup object was found, but the latest backup timestamp could not be extracted from the returned object.
```

## Current behavior

This email task is report-only.

It does not perform:

- ServiceNow DTSK update
- DTSK state change
- Assignment update
- CR Required field update
- work_notes update
- work_notes marker/idempotency check

## Future enhancement backlog

These items are intentionally not active yet because the required ServiceNow parameters/permissions are not available or not fully validated.

### 1. Update backup status on DTSK

Future enhancement should update the DTSK with the backup validation result after the report is generated.

Required before implementation:

- Confirm exact ServiceNow field name for backup status
- Confirm allowed values
- Confirm update permission
- Confirm whether update is allowed for all DTSK states

### 2. Change DTSK state

Future enhancement should change the DTSK state after backup validation.

Required before implementation:

- Confirm exact target state value
- Confirm state transition rules
- Confirm whether state change requires additional mandatory fields

### 3. Assign DTSK to shift owner

Future enhancement should assign the DTSK to a person in the current shift.

Required before implementation:

- Confirm shift owner source
- Confirm ServiceNow user/sys_id mapping
- Confirm assignment field names
- Confirm fallback owner when shift data is unavailable

### 4. Set CR Required to No

Future enhancement should set `CR Required` to `No` where applicable.

Required before implementation:

- Confirm exact field name
- Confirm accepted value for `No`
- Confirm whether field is writable through API
- Confirm whether business rules override this value

### 5. Fetch and update work_notes

Future enhancement should fetch existing work_notes, check for an existing workflow marker, and update work_notes only once.

Purpose:

- Add audit trail to DTSK
- Avoid duplicate updates on rerun
- Make workflow idempotent

Required before implementation:

- Confirm whether `work_notes` can be read directly from the DTSK table
- If not, confirm `sys_journal_field` access
- Confirm write permission to `work_notes`
- Confirm marker format

Suggested marker:

```text
[COHESITY_BACKUP_VALIDATION_WORKFLOW]
```

Suggested future idempotency logic:

```text
1. Fetch DTSK work_notes / journal entries
2. Search for marker + DTSK number + CI name
3. If marker exists, skip update
4. If marker does not exist, write work_notes
5. Continue to next DTSK
```

## Implementation rule for future writeback phase
