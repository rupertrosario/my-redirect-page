# 05 - team_servicenow_write_config

## Task type

ServiceNow Incident create/update task configuration.

This is not a JavaScript task.

## Position in workflow

```text
01_get_alerts
  ↓
02_validate_interfaces
  ↓
02a_search_ci_generic
  ↓
03_snow_search_team
  ↓
04_normalize_team_search
  ↓
create_team_incident  [loop]
update_team_incident  [loop]
```

## Purpose

Create or update Team incidents after `04_normalize_team_search.js` has classified each Team candidate.

Team incident identity is the cluster-level correlation ID generated by `02_validate_interfaces.js`:

```js
"DT_cohesity_ifdown_" + slug(clusterName)
```

Do not search or write by IP address, interface name, node ID, alert time, location, or Configuration item for Team incident identity.

## Inputs from previous tasks

`validate_interfaces` returns Team candidates:

```text
result("validate_interfaces").teamIncidents
```

Each Team candidate is the source item for both ServiceNow searches:

```text
02a_search_ci_generic  -> loops over result("validate_interfaces").teamIncidents
03_snow_search_team    -> loops over result("validate_interfaces").teamIncidents
```

This keeps the results attached by loop index:

```text
teamIncidents[0] -> CI search result[0] -> incident search result[0]
teamIncidents[1] -> CI search result[1] -> incident search result[1]
teamIncidents[n] -> CI search result[n] -> incident search result[n]
```

Do not chain the CI lookup from the incident search output, and do not chain the incident search from the CI lookup output. Both must use the same source list from `validate_interfaces`.

## Normalize responsibility

`normalize_team_search` should combine three aligned inputs:

```text
1. Team candidate from validate_interfaces.teamIncidents[index]
2. CI lookup result from search_ci_generic[index]
3. Incident search result from snow_search_team[index]
```

Expected routing:

```text
0 matching active incidents -> createTeamIncidents[]
1 matching active incident  -> updateTeamIncidents[]
2 matching active incidents -> noWriteTeamIncidents[]
```

`noWriteTeamIncidents[]` is manual-review only. Do not create or update when duplicates are found.

## Configuration item design

CI lookup happens before incident search in `02a_search_ci_generic`.

Preferred CI table:

```text
cmdb_ci_cohesity_cluster
```

Preferred CI search query:

```text
name={{ _.item.cluster }}^ORDERBYname
```

Fallback CI when no cluster CI is found:

```text
Cohesity (PRODUCTION)
```

Do not manually detach and reattach CI results in the ServiceNow create task. The CI value should be attached during normalization by matching the same loop index.

No JavaScript change is included in this document. Add `cmdb_ci` to the create payload only after the `normalize_team_search` change is reviewed and approved.

# Task: create_team_incident

## Task type

ServiceNow Create Record / Incident Create task.

## Loop configuration

Loop must be enabled.

Loop input / item list:

```text
result("normalize_team_search").createTeamIncidents
```

Condition:

```text
blank
```

Do not put `| length > 0` in the loop input. The loop input must be the array itself.

## Table

```text
incident
```

## Field mapping

Use the current loop item variable from Dynatrace:

```text
short_description = {{ _.item.short_description }}
description       = {{ _.item.description }}
correlation_id    = {{ _.item.correlation_id }}
comments          = {{ _.item.comment }}
```

Future approved CI mapping, after `normalize_team_search` emits `cmdb_ci`:

```text
Configuration item / cmdb_ci = {{ _.item.cmdb_ci }}
```

Do not add the `cmdb_ci` mapping until the normalize change is approved and present in the output.

## Recommended static fields

Set these only if required by the existing ServiceNow action/template:

```text
state             = New
assignment_group  = existing Team backup/hosting group used by the workflow
category          = Infrastructure
subcategory       = Backup
```

Do not add DC assignment groups to the Team create task.

## Expected create behavior

For each item in `createTeamIncidents[]`, create one active Team incident per Cohesity cluster.

The Team short description remains cluster-only:

```text
Cohesity Interface DOWN | <ClusterName>
```

IP/interface details stay in the description/comments, not the short description.

# Task: update_team_incident

## Task type

ServiceNow Update Record / Incident Update task.

## Loop configuration

Loop must be enabled.

Loop input / item list:

```text
result("normalize_team_search").updateTeamIncidents
```

Condition:

```text
blank
```

## Target record

Update by incident number returned from the ServiceNow search result:

```text
number = {{ _.item.number }}
```

If the Dynatrace ServiceNow action requires `sys_id` instead of `number`, update `04_normalize_team_search.js` first to include `sys_id`; do not guess the record ID in the write task.

## Field mapping

```text
comments       = {{ _.item.comment }}
correlation_id = {{ _.item.correlation_id }}
```

Do not overwrite short description during update unless explicitly required.

## Expected update behavior

For each item in `updateTeamIncidents[]`, append the latest interface-down details to the existing active Team incident.

---

# Safety rules

- GET-only applies to Cohesity calls only; ServiceNow writes are allowed only in these Team write tasks.
- Do not create or update when `noWriteTeamIncidents[]` contains duplicate active matches.
- Do not use lowercase `interface_alerts/` for this workflow.
- Do not modify `backup_failures/` from this Interface workflow.
